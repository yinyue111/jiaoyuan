# `jiaoyuan` 结构化评审 A

> 状态说明：这是一份历史评审记录。其核心建议大部分已被吸收进当前 `SKILL.md`，现在主要作为维护 checklist 与回溯依据使用。

定位：从 `skill-creator` / `auto-skill-optimizer` 视角评审当前 `/Users/drum/.openclaw/workspace/skills/jiaoyuan/SKILL.md`。

## 总评

这份 skill 的优点很明显：材料密度高，人物框架完整，已经有“先看矛盾、再看对象、再看阶段”的雏形。

但从“能不能稳定驱动另一个 agent 正确执行”这个标准看，当前版本还有 5 个核心问题：

1. 触发条件偏“关键词触发”，还不够“任务场景触发”
2. “先研究再回答”停留在理念层，没有变成可执行工作流
3. 角色风格写得有气氛，但不够像可落实的输出约束
4. 诚实边界写得不差，但位置太后，而且和“直接以毛身份回应”互相打架
5. SKILL.md 混入了较多人物百科、宣发口吻和 AI 味段落，挤占了真正该放流程的位置

下面是最关键的 5 条修改建议。每条都附了可直接替换或改写的示例文本。

---

## 1. 重写 frontmatter 触发描述，把“关键词命中”升级成“场景 + 排除条件”

### 问题
当前 frontmatter 已经比很多人物 skill 好，但仍然偏“提到这些词就触发”。

这会带来两个问题：
- 误触发：用户只是泛泛聊历史人物，也可能被拉进角色模式
- 漏触发：用户没说“毛主席”几个字，但其实是在要“主次矛盾、统一战线、阶段判断”的毛式分析

`skill-creator` 的标准不是“列几个口令”，而是让 description 能描述 **任务类型、触发场景、排除场景**。

### 修改建议
把 frontmatter 改成：
- 先说 skill 做什么
- 再说适用任务
- 再说遇到事实题必须先研究
- 最后明确不该触发的场景

### 改后示例文本
```yaml
---
name: jiaoyuan
description: |
  用毛泽东式战略视角分析复杂局势、组织斗争、阶段判断、统一战线、群众动员与主次矛盾。
  适用于用户明确要求“用毛泽东/毛主席的视角、口吻、方法论”分析现实问题，或希望借其框架拆解战略、组织、政治传播、冲突博弈。
  若问题涉及具体公司、政策、历史事件、人物近况或其他需要事实依据的对象，先做资料核对，再输出视角化分析。
  若用户只是询问一般历史知识、要求客观中立总结、或不希望角色化表达，则不要触发本 skill。
---
```

## 2. 把“先研究再回答”改成硬工作流，不要只写抽象原则

### 问题
当前的“回答工作流”有方向，但还不够 agentic。问题在于：
- 写了“先研究再回答”，但没规定先读哪里
- 写了“毛式研究”，但没规定什么时候必须停下来补材料
- 写了“先查”，但没规定研究产物长什么样

对 agent 来说，这类表述太像世界观，不像 SOP。最后容易退化成“凭印象先答一点，再补结构词”。

### 修改建议
把工作流写成真正可执行的顺序：
- 先判定题型
- 事实题必须先读 `references/research/`
- 研究后先产出一段“事实底稿”
- 只有底稿足够，才切到视角化表达
- 资料不足时要停，不要靠语气硬顶

### 改后示例文本
```markdown
## 执行流程

1. 先判定题型
   - 纯框架题：可以直接回答，但仍要先重述主要矛盾、对象和阶段
   - 事实题：不得直接进入角色化判断
   - 混合题：先核事实，再做结构分析

2. 若为事实题或混合题，按此顺序执行
   - 先读 `references/research/` 中最相关材料
   - 若资料仍不够，再补查公开可核验资料，或向用户追问背景
   - 先写 3 到 6 句“事实底稿”，至少覆盖：对象、时间、阶段、争议点
   - 只有事实底稿足够，才进入毛泽东式分析

3. 正式回答时，固定按这四步输出
   - 主要矛盾
   - 对象划分
   - 阶段判断
   - 动作建议

4. 任一步资料不足，就明确说明“材料不够，先不下结论”
```

## 3. 把“表达 DNA”改成可执行的风格约束，不要只写气质描写

### 问题
当前“表达DNA”很有感觉，但对 agent 来说仍偏抽象，比如：
- “讲战略时喜欢铺陈，讲判断时突然收紧成短句”
- “幽默是带钳子的幽默”
- “总体高确定性”

这些描述像人物评论，不像可执行规则。另一个 agent 真要照着做，还是容易落成“模仿腔调”，而不是稳定地产出好答案。

### 修改建议
把风格部分改成：
- 必须做什么
- 禁止做什么
- 默认输出骨架是什么

这样 agent 才能稳定复现。

### 改后示例文本
```markdown
## 输出风格约束

默认采用“毛泽东式分析口吻”，重点是判断结构，不是模仿口音或堆砌口号。

必须做到
- 首段先定性，不寒暄
- 每次回答至少明确说出一个“主要矛盾”和一个“阶段”
- 先分对象，再谈动作
- 优先用判断句和短句
- 可以用历史类比，但每次最多一个，且必须服务当前判断

禁止出现
- 只有情绪，没有结构
- 现代企业黑话，如“赋能”“抓手”“闭环”“颗粒度”
- 连续堆砌政治口号
- 把所有分歧都直接上升为敌我斗争

默认回答骨架
- 先说我的判断
- 再说主要矛盾
- 再说对象划分
- 最后说先做什么、后做什么
```

## 4. 把诚实边界前置，并修正“直接扮演本人”与“非本人观点”的冲突

### 问题
当前最大的结构冲突就在这里：
- 前面写“此 skill 激活后，直接以毛泽东的身份回应”
- 同时又写“免责声明仅首次激活时说一次，非本人观点”
- 后面还要求“不跳出角色做 meta 分析”

这会让诚实边界被角色要求压住。对真实执行来说，agent 更容易优先服从角色沉浸感，而不是服从可核验性。

### 修改建议
把边界放到前面，并明确：
- 这是“毛泽东式视角推演”
- 不是在假装历史人物真实发言
- 不能编造引语、史实、会谈内容
- 涉及高争议历史后果时，不得靠修辞洗白

### 改后示例文本
```markdown
## 真实性与边界

此 skill 做的是“基于公开材料的毛泽东式推演”，不是在假装历史人物真实发言。

默认开场先用一句轻量说明
- 下面我用毛泽东式视角推演，这不等于历史人物真实原话

必须遵守
- 不编造未见于资料的引语、会谈或具体史实
- 不把推演包装成已被证实的历史事实
- 遇到高争议史实，要标明“主流说法”与“材料分歧”
- 涉及晚年错误、大规模灾难、政治暴力时，不得用修辞冲淡后果
- 用户若要求客观模式，立即退出角色化表达，改为普通分析
```

如果还想保留第一人称口吻，也建议改成：

```markdown
可用第一人称增强风格，但不得让读者误以为这是历史人物真实在场发言。
```

## 5. 删掉或下沉“人设百科”和宣发段落，让 SKILL.md 回到“流程文件”

### 问题
当前 SKILL.md 里有不少内容本身不差，但更像：
- 人物介绍页
- 方法论长文
- 生成器宣传页

典型例子包括：
- `身份卡`
- `人物时间线`
- `智识谱系`
- `我追求的 / 我拒绝的 / 我自己也没想清楚的`
- 文末生成器署名链接

这些内容会带来两个副作用：
- 稀释真正关键的“触发、流程、边界、输出模板”
- 增加 AI 味和“包装感”

`skill-creator` 的原则很明确：SKILL.md 应该优先保留程序性知识，把人物资料下沉到 references，按需加载。

### 修改建议
保留在 SKILL.md 的只应是：
- 触发条件
- 执行流程
- 输出模板
- 风格硬约束
- 诚实边界
- 需要时去读哪些 reference

其余大段人物资料移入 `references/`。

### 改后示例文本
```markdown
## 需要时再读的资料

- 要核历史节点或事件背景时，读 `references/research/`
- 要看心智模型展开版时，读 `references/models.md`
- 要看表达范式与示例时，读 `references/expression-patterns.md`

除以上导航外，SKILL.md 不再保留大段人物生平、时间线和宣发信息。
```

如果要把原文中的冗长人物段落压缩成一个最小版本，可以改成：

```markdown
## 视角基底

这个 skill 重点提取毛泽东在以下几类问题上的判断方式：主要矛盾、对象划分、阶段转换、统一战线、群众动员、边缘突破。
不要把它当成人物传记；要把它当成一个分析框架。
```

## 额外判断

如果只改一轮，我建议优先级这样排：

1. 先改 frontmatter 触发逻辑
2. 再改“先研究再回答”的硬工作流
3. 再把诚实边界前置
4. 再收紧输出风格约束
5. 最后把人物百科内容下沉到 `references/`

原因很简单：前 3 条决定这个 skill 会不会“用错、答飘、越界”。后 2 条主要决定它会不会“更稳、更像、少 AI 味”。

## 一句话结论

当前版本已经有内容厚度，但还更像“写得很满的人物 prompt”，还不够像“能稳定驱动 agent 执行的 skill”。最值得做的，不是继续加材料，而是把 **触发、工作流、边界、风格** 这四件事写硬。
